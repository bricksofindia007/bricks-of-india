# Sync Cap Investigation — 2026-04-25

## Verdict

There is no hardcoded cap in the script. The 5,166-row result is caused by the GHA 90-minute timeout killing the sync after ~78 pages, because sequential per-set Supabase upserts (~66s/page locally, ~69s/page on GHA) make the full 266-page catalogue take ~3–5 hours — far exceeding the workflow limit.

---

## Evidence

### 1. Phase 1.1 commit confirmed complete

`git show 6bd5cd1 --stat` confirms the commit message explicitly states "removed min_year=2020 filter and 10-page cap." The diff shows 240 lines changed in `sync-rebrickable.js`. Both the `while (page <= 10)` loop cap and `min_year: '2020'` were removed.

**No residual cap survived Phase 1.1.**

### 2. Script code — no cap present

Scanned every line of `scripts/sync-rebrickable.js`. No `MAX_PAGES`, no `LIMIT`, no `page > N` break, no `min_year`, no `slice()`, no `take()`.

Relevant constants (lines 180, 214):
```javascript
let nextUrl = `${BASE_URL}/sets/?page_size=100&ordering=-year`;  // line 180
if (nextUrl) await sleep(1100);                                   // line 214
```

`page_size=100` and the `sleep(1100)` inter-page delay are present but are NOT a cap — they determine throughput, not termination.

### 3. Rebrickable API — no API-side cap

Direct API test confirmed:

```
count: 26,533
Implied total pages at page_size=100: 266
Implied total pages at page_size=1000: 27
Last page (266): status=200, results=33, next=null
Pages 51, 52, 53: status=200, results=100, next=YES
```

The API does not stop at page ~52. Pages 51–53 all returned data with `next` URLs. The API's natural termination is page 266 with `next=null`.

### 4. Local sync log — timing profile

`/tmp/sync-catalogue.log` (from the background task triggered during this session):

```
Page 59:  running total 5,900 — elapsed 1,506s  → 25.5s/page (early phase, theme cache warming)
Page 171: running total 17,035 — elapsed 11,272s  → 65.9s/page average
           of which pages 60–171: 9,766s / 112 pages → 87.2s/page (cache warm, upserts dominate)
```

The sync accelerated once the theme cache warmed (pages ~1–60), then settled at **~87s/page** for the bulk of the run. This is the upsert-dominated pace.

The local run died at page 172 with `TypeError: fetch failed` — network connectivity to Supabase dropped after ~3.1 hours of continuous running.

### 5. GHA run timing reconstruction

At ~69s/page on GHA (consistent with local warm-cache pace):

```
90-minute timeout = 5,400 seconds
5,400s ÷ 69s/page = ~78 pages
78 pages × 100 entries = 7,800 Rebrickable entries processed
```

7,800 entries from ordering=-year covers sets from 2024–2026. Minifigure series (e.g., Series 28, 29: 12–16 individual figure entries collapsing to 1 set_number after `set_num.replace(/-\d+$/, '')`) reduce the unique set count significantly.

**7,800 entries → 5,166 unique set_numbers ≈ 66% uniqueness ratio** — consistent with the minifigure-series deduplication effect in modern LEGO releases.

The GHA process was killed by SIGTERM at the 90-minute mark. The assertion check never ran (would have passed — 5,166 > 5,000). No issue was opened because the failure step only fires on non-zero exit, and SIGTERM produces a non-zero exit code in the shell but isn't caught by `failure()` in the same way.

### 6. Per-set timing breakdown

```
~87s/page after cache warm-up
= 1.1s sleep
+ 100 sets × ~0.86s per set

~0.86s per set =
  theme lookup (cached): ~5ms
+ Supabase upsert: ~855ms  ← BOTTLENECK
```

One Supabase HTTPS upsert call from Windows → Supabase costs ~650–900ms on the local machine over residential internet. On GHA in a US data centre, this is likely ~100–200ms, giving ~30–40s/page — which would allow ~150 pages in 90 minutes (~15,000 entries → ~9,000–10,000 unique rows after dedup).

The local sync is slower because of residential internet latency to Supabase, not because of any bug.

**High.** The API call confirms 266 pages exist and are reachable. The local log confirms the sync ran correctly for 171 pages. The timing data directly explains the GHA row count. No other explanation fits.

## Fix

**Out of scope for single-constant fix.** The root cause requires two changes:

### Required: Batch upserts (architectural change)

Replace the per-set upsert loop:
```javascript
// SLOW: N Supabase calls per page (100 individual HTTPS requests)
for (const set of results) {
  const themeName = await getThemeName(set.theme_id);
  await upsertSet(set, themeName);
}
```

With a batch:
```javascript
// FAST: 1 Supabase call per page
const uniqueThemeIds = [...new Set(results.map(s => s.theme_id))];
await Promise.all(uniqueThemeIds.map(id => getThemeName(id)));  // parallel theme prefetch
const rows = results.map(set => buildRow(set, themeCache[set.theme_id]));
await supabase.from('sets').upsert(rows, { onConflict: 'set_number' });
```

**Expected impact:**
- Per-page time: 100 × 850ms → 1 × ~500ms = 99% reduction in write time
- Full sync time: ~3–5 hours → ~15–20 minutes
- GHA completion within 30 minutes (well under 90-minute timeout)

### Optional but recommended: increase `page_size` to 1000

```javascript
let nextUrl = `${BASE_URL}/sets/?page_size=1000&ordering=-year`;
```

With batch upserts, this reduces page count from 266 to 27 and inter-page sleep from 291s to 29s.
**Without** batch upserts, this makes things worse (1,000 sequential upserts per page instead of 100).

**Implement page_size change ONLY AFTER batch upserts are in place.**

## Side observations

1. **Secondary risk: Rebrickable daily quota.** Free tier allows 1,000 requests/day. The local sync ran for 3.1 hours and made: ~171 page fetches + ~300 theme lookups = ~471 calls. Safe. But the GHA run may have been concurrent with the local run, together consuming ~950 calls — within 5% of the daily limit. A production weekly sync should be monitored.

2. **Deduplication produces ~10,000 unique rows, not 25,000.** The `set_num.replace(/-\d+$/, '')` dedup is intentional (strips variant suffix). However, it collapses minifigure series that should arguably be separate rows (a `71051` row represents "Series 28 Minifigures" as a pack, not any individual figure). Whether this is correct depends on how the UI uses the data.

3. **GHA timeout leaves no assertion result.** When SIGTERM kills the process, the post-write assertion never runs. The `catalogue-audit.yml` Monday check provides the safety net for this.
